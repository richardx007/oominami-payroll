## 1. システム概要

アルバイト従業員の勤務時間・交通費申告をもとに給与を計算し、給与明細の配信および税理士向け資料の作成・送付までを行う、小規模事業者向けのシンプルな給与管理システム。

### 業務フロー(月次サイクル)

```
バイト: 勤務日・交通費をスマホで随時入力
   │
   ▼ 25日締め
管理者: 入力状況確認 → 未入力者へ催促 → 締め処理
   │
   ▼
システム: 給与計算(時給×時間 + 交通費 + 昼食補助 − 所得税)
   │
   ▼ 月末払い
管理者: 給与明細をメール配信(個別/一斉)
   │
   ▼
管理者: 税理士向け資料(PDF)を確認・メール送信
```

### スコープ外

- 年末調整、社会保険・雇用保険の管理
- 銀行振込連携(支払いは手動)
- 正社員給与(月給制)

## 2. 技術スタック

| 項目 | 採用技術 | 理由 |
|------|---------|------|
| フロントエンド | **Next.js (App Router) + TypeScript** | Cloudflare Pages/Workers で動作、レスポンシブ対応が容易 |
| ホスティング | **Cloudflare Pages + Workers** (`@opennextjs/cloudflare`) | 指定要件。無料枠: 10万リクエスト/日 |
| DB / 認証 | **Supabase** (PostgreSQL + Auth) | 指定要件。無料枠: 500MB DB、月5万MAU |
| UI | **Tailwind CSS + shadcn/ui** | モダンですっきりしたデザイン、モバイルファースト |
| メール送信 | **Resend**(無料枠 100通/日)または Gmail SMTP | 明細配信・催促・一斉報知・税理士送付 |
| PDF生成 | **react-pdf / @react-pdf/renderer**(Workers上で生成) | 給与明細・税理士資料のPDF化 |
| バリデーション | Zod | フロント/API共通のスキーマ検証 |

### 既存ボトル管理システムとの同居について

同一 Supabase アカウント・Cloudflare アカウントで運用するため:

- **Supabase**: 無料枠はプロジェクト2つまで作成可能 → **別プロジェクトとして新規作成を推奨**(スキーマ・認証・バックアップが完全分離され安全)。もし既にプロジェクト数上限なら、既存プロジェクト内に `payroll` スキーマを切って同居させる(RLSとスキーマ分離で干渉を防止)。
- **Cloudflare**: Pages プロジェクトは無料で複数作成可能 → 別プロジェクト・別サブドメインでデプロイ。Workers 無料枠(10万req/日)はアカウント合算のため、両システム合計でも小規模利用なら十分余裕。
- 無料枠の主な注意点: Supabase 無料プロジェクトは**1週間アクセスがないと一時停止**する → 月次のヘルスチェック(Cloudflare Cron Trigger で定期ping)を入れて回避。

## 3. データベース設計(案)

```
employees(雇用者)
  id, employee_no(雇用者No), name, email, auth_user_id(Supabase Auth連携),
  is_admin, status(在籍/退職), created_at

wage_rates(時給履歴)※値上げ対応
  id, employee_id, hourly_wage, effective_from(適用開始日)

allowance_settings(手当設定)
  id, lunch_allowance_per_day(昼食補助/日), effective_from

work_entries(勤務表)
  id, employee_id, work_date, start_time, end_time, break_minutes,
  transport_cost(交通費), note, created_at, updated_at
  ※ pay_period との一意制約(同一日重複防止)

pay_periods(給与計算期間)
  id, period_label(例: 2026年7月分), start_date(前月26日), end_date(当月25日),
  payment_date(月末), status(入力中/締め済み/支払済み)

payslips(給与明細)
  id, employee_id, pay_period_id, work_days, total_hours,
  base_pay(基本給), transport_total, lunch_total,
  gross_pay(総支給), income_tax(源泉所得税), net_pay(差引支給額),
  finalized_at, emailed_at

notifications(連絡・催促・一斉報知)
  id, sender_id, recipient_id(null=全員), type(個別/一斉/催促),
  subject, body, sent_at

tax_reports(税理士向け資料)
  id, pay_period_id, pdf_url, emailed_to, emailed_at
```

### 主な設計ポイント

- **時給の値上げ対応**: `wage_rates` に適用開始日つきで履歴を持ち、勤務日ごとに有効な時給を引いて計算。過去分の再計算にも耐える。
- **所得税**: 源泉徴収税額表(月額表・乙欄/甲欄)に基づく簡易計算ロジックをテーブル化して保持。扶養親族数は `employees` に持たせる。税額表は年度更新可能な設計にする。
- **締め処理**: `pay_periods.status` が「締め済み」になったら該当期間の `work_entries` を編集ロック(RLSで制御)。
- **RLS(行レベルセキュリティ)**: バイトは自分のレコードのみ読み書き可、管理者は全件可。Supabase RLS で担保。

## 4. 画面設計(案)

### バイト向け(スマホ基本)

| 画面 | 内容 |
|------|------|
| 初回登録 | メールアドレスで招待リンク受領 → パスワード設定(Supabase Auth の invite フロー) |
| ログイン | メール+パスワード |
| 勤務表入力 | カレンダー型UI。日付タップ → 出退勤時刻・休憩・交通費を最小タップで入力。前回値のコピー機能 |
| 月間サマリ | 当月の勤務日数・合計時間・交通費合計を一覧表示 |
| 給与明細閲覧 | 過去明細の一覧・詳細表示 |
| お知らせ | 管理者からの連絡・一斉報知の閲覧 |

### 管理者向け(レスポンシブ: スマホ/タブレット/PC)

| 画面 | 内容 |
|------|------|
| ダッシュボード | 当期の入力状況一覧(誰が何日分入力済みか、未入力者のハイライト) |
| 雇用者管理 | 登録(招待メール送信)・時給設定(適用開始日つき)・退職処理 |
| 勤務表確認 | 個人別・日別の申告内容確認、必要時の代理修正 |
| 連絡・催促 | 個別メッセージ/入力催促/全員一斉報知の送信(アプリ内+メール) |
| 締め処理 | 25日締め実行 → 給与計算プレビュー → 確定 |
| 明細配信 | 給与明細PDFの個別/一斉メール配信 |
| 税理士向け資料 | 支給一覧・源泉徴収額の集計画面、PDF出力、税理士宛メール送信 |
| 設定 | 昼食補助単価、税理士メールアドレス、会社情報 |

### UI/UX方針

- モバイルファースト。勤務表入力は「タップ数最小」を最優先(既定勤務時間のプリセット、前回交通費の自動補完)
- shadcn/ui ベースのすっきりした白基調 + アクセントカラー1色
- 入力必須項目のみ表示し、詳細はアコーディオンに畳む

## 5. 開発フェーズ計画

### フェーズ0: 基盤構築(〜3日)
- リポジトリ初期化(Next.js + TypeScript + Tailwind + shadcn/ui)
- Supabase プロジェクト作成、スキーマ・RLS定義(マイグレーション管理)
- Cloudflare Pages デプロイパイプライン(GitHub連携で自動デプロイ)
- Supabase Auth 設定(メール招待+パスワード設定フロー)

### フェーズ1: 認証・雇用者管理(〜4日)
- 管理者による雇用者登録・招待メール送信
- バイト初回登録(パスワード設定)・ログイン
- 時給設定(履歴つき)、昼食補助設定

### フェーズ2: 勤務表入力(〜5日)★バイト向けコア
- スマホ向けカレンダー入力UI(勤務日・時刻・休憩・交通費)
- 月間サマリ表示
- 管理者の入力状況ダッシュボード

### フェーズ3: 給与計算・締め処理(〜5日)★計算コア
- 給与計算エンジン(時給履歴適用、昼食補助、交通費、源泉所得税)
- 締め処理(25日締め・編集ロック・再計算)
- 計算ロジックのユニットテスト(税額計算は特に重点)

### フェーズ4: 明細・通知・配信(〜5日)
- 給与明細PDF生成・画面閲覧
- メール配信(個別/一斉)— Resend 連携
- 連絡・催促・一斉報知機能

### フェーズ5: 税理士向け資料(〜3日)
- 月次支給集計画面(支給総額・源泉徴収税額一覧)
- PDF出力・税理士宛メール送信

### フェーズ6: 仕上げ(〜3日)
- E2Eテスト(締め〜配信の一連フロー)
- Supabase 休止対策の Cron ping、エラー監視
- 実データでの試行運用(1か月分の並行稼働を推奨)

**合計目安: 約4週間**(1フェーズごとに動くものを確認しながら進めるインクリメンタル開発)

| リソース | 無料枠 | 想定使用量 | 判定 |
|---------|-------|-----------|------|
| Supabase DB | 500MB | 数MB/年 | ◎ |
| Supabase Auth | 5万MAU | 〜15名 | ◎ |
| Cloudflare Workers | 10万req/日 | 数百req/日(ボトル管理と合算でも余裕) | ◎ |
| Resend メール | 100通/日, 3,000通/月 | 明細一斉配信でも月数十通 | ◎ |

## 7. リスクと対策

| リスク | 対策 |
|-------|------|
| 源泉徴収税額の計算誤り | 税額表をデータ化しテスト網羅。初月は手計算と突合。税理士確認を挟む |
| Supabase 無料プロジェクトの自動休止 | Cron Trigger による定期ping |
| メール不達(明細・招待) | 送信ログを保持し、管理画面で配信状態を確認できるようにする |
| 締め後の申告漏れ | 管理者による締め解除→再計算フローを用意 |
| ボトル管理システムとのリソース競合 | 別プロジェクト分離を基本とし、Workers リクエスト数を月次で確認 |
